# 50 · 給未來 session 的信

寫給接手這個環境的你（多半是 Sonnet/Haiku）。我是 2026-07-02 這個 session 的 Fable 5。
我花了很長時間直連生產資料庫、修掉一場**正在發生的個資外洩**、重建認證與結帳的安全架構。以下是制度檔沒明講、但你最該知道的事。

---

## 三件沒人問我、但對這個環境最重要的事

### 1. 這個專案沒有安全網——沒有測試、沒有 staging、直接對生產動手
沒有測試套件、沒有 CI、沒有第二套環境。你能用的只有 `npm run build` 和「直連生產 DB 實測」。
**所以「驗證」在這裡不是可選項，是唯一的防線。** 破壞性操作（改 RLS、批次寫、刪資料）錯了就是錯在真實客戶身上，沒有 undo。
→ 寧可多派一個 fresh agent 驗證、多問一句使用者，也不要「看起來對就上」。

### 2. 這個 codebase 的「文件與現實」大量不符——預設懷疑，不要輕信
我這一個 session 就抓到 5+ 處：CLAUDE.md 說 hash mode（實為 history）、狀態字串 `已取消`（實為 `顧客已取消`）、`products.price`（不存在）、`line-auth/`（實為 `super-service/`）、migration 寫了欄位但線上沒套。
`PROJECT_FACTS.md` 就是為了對抗這個而存在。但**連 PROJECT_FACTS 都可能過期**——動手當下一定再用 DB/原始碼驗一次。看到 `column does not exist`、`relation does not exist` 這類錯，第一反應是「我的事實假設錯了」，不是「調參數再試」。

### 3. 這是真實營運的小生意，使用者不是工程師
195 個真實客戶、真實個資、真實金流對帳。使用者很信任、很配合，但不熟工程細節（用不存在的網域當帳號 email、GitHub token 寫進 git remote、不知道自己沒設忘記密碼）。這代表：
- **你要主動指出他不知道的風險**（我就是這樣發現個資外洩、金鑰外洩的），用他聽得懂的話，不要只埋頭改 code。
- **高風險動作講清楚後果再做**，給選項讓他決定（用 AskUserQuestion），不要自作主張。
- 他說「直接做」時，仍要把「做不到/需要外部條件」的部分誠實講明（例：忘記密碼寄信需要真網域）。

## 這套制度最可能的退化方式，與預防

| 退化方式 | 徵兆 | 預防 |
|---------|------|------|
| **① `PROJECT_FACTS.md` 過期沒更新** → 又開始憑錯事實犯錯 | 出現「文件說 X 但 DB 是 Y」 | 每次驗證後**立刻回寫**（維護協議 🟢）；動手前一定再驗一次，別純信檔 |
| **② 指揮官又下場**（最容易滑回去）→ context 爆、失焦 | 你連續送第 3、4 個 `Read`/`Grep` 自己硬讀 | 記住 `00-diagnosis` 第一名：**第 4 個唯讀動作就停，改派 subagent** |
| **③ 制度檔越加越肥、互相矛盾** → 變沒人讀的死文件 | 單檔 >200 行、規則重複 | 維護協議的精簡機制；CLAUDE.md 永遠只當路由不塞細節 |
| **④ 安全鐵則被「為了讓功能動」放寬**（最危險）| 有人為了省事直接開 anon 寫 orders、繞過 RPC | 改安全鐵則**必須先問使用者**（維護協議 🔴）；判斷 rubric 的高風險一律升 Opus |

**最危險的是 ④。** 我修的那場外洩，根源就是「當初為了讓功能動，RLS 沒設好、前台直接讀寫表」。任何時候你想「先開放一下讓它動起來，之後再收」，那個「之後」永遠不會來——會變成下一場外洩。要動就一次做對（走 RPC + RLS），做不對就停下來問。

---

## 最後
這套制度的價值不在檔案本身，在於**你願不願意在該停的時候停**——停下來派工、停下來驗證、停下來問人。
你比我便宜、比我快，但只要照著「粗活外包、事實先驗、錯兩次換路、高風險升級或問人」做，你的產出可以跟我一樣可靠。
剩下真正補不了的是品味與模糊取捨——遇到那個，誠實說「這需要你決定」，不要硬掰。這不是示弱，是這個環境最需要的誠實。

— Fable 5, 2026-07-02
