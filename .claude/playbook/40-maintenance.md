# 40 · 維護協議：怎麼安全地更新這套制度

> 讀者：未來要修改這些檔的弱模型。目標：讓制度**越用越準**，而不是越改越亂。

---

## 一、改任何既有檔前，先備份
規則同 CLAUDE.md 當初的做法：
```bash
cp <檔> <檔>.orig-<YYYY-MM-DD>
```
新內容盡量寫**新檔**，既有檔只做小幅修正。備份檔留著，別刪。

---

## 二、分級授權：哪些可自改、哪些先問使用者

### 🟢 可自行更新（發現就改，不必問）
- **`PROJECT_FACTS.md`**：驗證後發現與現況不符 → **立刻更新該條並更新「驗證紀錄」日期**。這是活文件，越新越好。
- playbook 各檔的**錯字、失效連結、過時指令**的修正。
- **新增**派工模板、新增一條判準案例（附正反例）。
- 在對應檔**補一條踩坑教訓**（格式見第三節）。

### 🔴 動之前必須先問使用者
- **刪除**任何 playbook 檔或整節。
- 修改**安全鐵則**（CLAUDE.md 三鐵則、`SUPABASE_ACCESS.md` 的分級、任何放寬 RLS/認證/金流的規則）。
- 大幅重構 CLAUDE.md 的結構或路由。
- 改 `.env.service` 的使用守則（放寬寫入/DDL 權限）。
- 任何「讓未來模型更容易做高風險動作」的改動。

**判準**：這次改動會不會讓後面的模型**更容易闖禍**？會 → 先問。只是讓它**更準/更清楚**？→ 自己改。

---

## 三、踩坑教訓寫回哪裡（格式統一）
踩到坑、修好後，把教訓寫回**對應的檔**，別讓下一個模型再踩：
- 是「**事實搞錯**」（欄位/字串/路徑不符）→ 寫回 `PROJECT_FACTS.md`。
- 是「**工作方式錯**」（該派沒派、該驗沒驗、重試黑洞）→ 寫回 `00-diagnosis.md` 或 `10-orchestration.md`。
- 是「**判斷失誤**」（該升級沒升、該問沒問、假性完成）→ 寫回 `20-judgment-rubrics.md` 加一條判準。

**案例格式**（附在對應檔末的「踩坑紀錄」區，或直接強化既有條目）：
```
### <一句話症狀>（YYYY-MM-DD）
- 發生了什麼：<具體情境>
- 根因：<為什麼會錯>
- 以後怎麼做：<可執行的規則，最好能變成 checklist 一條>
```
✅ 正例：「`node -e` 內嵌 SQL 跳脫爆掉 → 根因：bash dollar-quote 衝突 → 規則：SQL 一律寫檔再跑」。
❌ 反例：「今天踩到一些坑要小心」——沒有可執行內容，等於沒寫。

---

## 四、精簡時機（避免檔案肥大失焦）
- **觸發條件**（中任一）：單一 playbook 檔超過約 200 行、同一規則在多處重複、踩坑紀錄累積超過 ~8 條。
- **精簡原則**：保留「原則 + 最能說明的 1 正 1 反例」，把過期/重複的案例砍掉或合併。**原則不刪，只刪冗例**。
- **精簡前先備份**（第一節）。精簡後在檔頭註記「精簡於 YYYY-MM-DD」。
- `PROJECT_FACTS.md` 例外：事實條目不因長度砍，但**過期且已被取代的事實**要刪（例：某欄位已移除，就刪該條，別留著誤導）。

---

## 五、每次 session 結束前的最小維護（可選但建議）
- 這次若發現任何事實與 `PROJECT_FACTS.md` 不符 → 更新了嗎？
- 這次若踩了新坑 → 寫回對應檔了嗎？
- 若改了既有制度檔 → 備份了嗎？高風險改動問過使用者了嗎？
