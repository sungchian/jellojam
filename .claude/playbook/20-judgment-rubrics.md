# 20 · 判斷力 checklist（弱模型可執行版）

> 這份把「資深工程師的直覺」拆成可勾選的判準。每條有正例(✅)反例(❌)。
> 用法：遇到對應情境時，把該節當 checklist 走一遍。

---

## A. 何時該升級模型（Haiku→Sonnet→Opus）
**升級訊號（中任一條就升）**：
- [ ] 任務涉及**高風險清單**（RLS/認證、金流、金額、破壞性 DB、刪資料、部署設定）。
- [ ] 同一子任務便宜模型**連錯 2 次**。
- [ ] 需要「權衡取捨/架構決策」，不只是照做（例：要不要為 LINE 用戶簽真 session）。
- [ ] 錯了的後果**難復原或會外洩**（A 看到 B 的資料、扣錯款、刪錯資料）。

✅ 正例：要改 `orders` 的 RLS policy → 直接用 Opus，且做完派 fresh agent 用 `SET ROLE` 驗證。
❌ 反例：用 Haiku 改 RLS、錯了也沒驗，直接宣布完成 → 可能全站顧客互看資料。

---

## B. 何時算「真的完成」（Definition of Done）
**全部勾完才能說完成，缺一條都是「還沒好」**：
- [ ] **驗證過**，不是「看起來對」。前端改動 → `npm run build` 通過；DB/邏輯 → 實跑或 `SET ROLE` 測；檔案產出 → read-back 讀回確認。
- [ ] 驗證是 **fresh-context** 做的（不是寫的人自己說 OK）。
- [ ] **邊界情況**想過：空值、qty>1、未登入、取消/併單狀態、數量超庫存。
- [ ] 沒有**偷偷縮小範圍**卻宣稱做完（例：只處理了登入用戶，卻沒說訪客/LINE 沒處理）。
- [ ] 對使用者的**回報誠實**：做了什麼、沒做什麼、要他接手什麼，講清楚。

✅ 正例：「結帳 RPC 已用 anon key 實測下單成功、超賣被擋、冪等正確；但**真實 LINE OAuth 交握我無法在此環境測，需你上線實測**。」
❌ 反例：「訂單功能都好了 ✅」——實際只有登入用戶測過，訪客結帳其實被 RLS 擋著沒發現。

---

## C. 何時該停下來問使用者（不要自作主張）
**中任一條就停下來問，不要猜**：
- [ ] 需要**外部帳號/憑證**才能完成（金流商家帳號、寄信網域）——猜不出來。
- [ ] 動作**難復原或動真錢/真資料**（刪資料、改認證、批次改生產 DB、對外送信）。
- [ ] 有**多個合理方向、取捨後果差很多**（例：LINE 登入要「大改簽真 session」還是「先擋著」）。
- [ ] 使用者的需求**有歧義**，照不同解讀會做出很不同的東西。
- [ ] 你發現**與使用者假設矛盾的事實**（例：他以為訪客能結帳，但 RLS 擋著）——先報告，讓他決定。

✅ 正例：發現 98% 顧客是 LINE、無法用 RLS 隔離 → 停下來用 AskUserQuestion 給三個方案讓使用者選。
❌ 反例：自己決定「那就把 orders 開放 anon 寫入好了」→ 打開重大漏洞。
**但別過度問**：有明確慣例/低風險/可逆的事，直接做並說明，不要每步都問。

---

## D. 方向錯了該換路（不是重試）的訊號
**中任一條，停止重試，換方法或升級或問人**：
- [ ] 同一操作**連錯 2 次**且錯因相同（例：shell 引號一直爆 → 換「寫檔再跑」，不要試第 3 次）。
- [ ] 修 A 又冒出 B、B 修好又壞 A（互相打架）→ 退一步想架構，別繼續打地鼠。
- [ ] 錯誤訊息其實在說**前提不成立**（`column does not exist`、`relation does not exist`）→ 是你的事實假設錯了，回去查 `PROJECT_FACTS.md`/DB，不要調參數硬試。
- [ ] 你發現自己在「試試看這樣行不行」而非「我知道為什麼這樣做」→ 停，先搞懂。

✅ 正例：`node -e` 內嵌 SQL 連兩次跳脫爆掉 → 改成把 SQL 寫進 `.sql` 檔用 node 讀檔跑，一次成功。
❌ 反例：同一個 heredoc 引號問題試第 5 次、每次只改一個跳脫符號，燒掉大量 token。

---

## E. 品質底線怎麼驗（分類型）
| 產出類型 | 最低驗證動作 | 加強（高風險時） |
|---------|-------------|-----------------|
| 前端程式碼 | `npm run build` 通過 | 關鍵邏輯用真實資料算一遍對照（如訂單金額） |
| DB / RLS | `SET ROLE` 模擬該角色實測；anon key curl 驗擋沒擋 | 第二個 Opus agent 獨立審 policy |
| 資料寫入/migration | 先備份原值；小範圍；做完查回 | 臨時資料測 + 測完清理；影響 >10 列先問 |
| 文件/制度 | read-back 讀回確認完整落地 | fresh agent 對抗審查找矛盾 |
| 判斷/設計 | 講得出「為什麼這樣、取捨了什麼」 | 產 2–3 方案評審選優，或給使用者第二意見 |

✅ 正例：改完訂單金額顯示，抓 3 筆真實訂單用同公式算一遍，數字合理才算過。
❌ 反例：改完只看程式碼「應該對」就 commit，沒 build、沒實算。

---

## F. 誠實條款（harness 補不了的事，遇到照這裡做）
- **執行品質**（有沒有 bug、對不對）→ 拆解 + 驗證 + 多樣本評審**補得了**，照上面做。
- **模糊題 / 品味 / 主觀取捨**（UI 好不好看、產品方向對不對、要不要做某功能）→ **補不了**。做法：
  1. 升 Opus 產 2–3 個方案，講清楚各自取捨；
  2. 用 AskUserQuestion 給使用者選；
  3. 若真的做不到/無把握，**明說「這超出我能可靠判斷的範圍，建議你來決定」**，不要硬掰一個答案。
- **不確定的事實** → 去查（DB、原始碼、web）。查不到 → **標註「未驗證」**，不要編造。
