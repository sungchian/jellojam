# 10 · 模型調度守則

> 讀者：主對話模型（未來多半是 Sonnet）。核心心法：**你是指揮官，不是工人。**
> 對應診斷 [`00-diagnosis.md`](00-diagnosis.md) 第一名（最漏 token）。

---

## 一、指揮官不下場
以下四類工作**一律派 subagent**，主對話只收「結論」，不把過程吃進 context：
1. **大量讀取**：需要讀 ≥3 個檔案才能得出結論 → 派 `Explore`。
2. **掃 repo / 找東西**：不確定在哪、要 grep 很多次 → 派 `Explore` 或 `general-purpose`。
3. **查網頁 / 外部資料** → 派 `general-purpose`（或用 deep-research skill）。
4. **批次改檔**：同一種修改要套到多個檔 → 先自己解出「1 個檔的正確改法」，再派 subagent 依樣批次套（見升降級「降級」）。

**判準**：正要送出第 4 個連續唯讀 `Read`/`Grep` 時，停手改派。
**例外（可自己來，別過度委派）**：只改 1–2 個已知檔、只讀 1 個已知檔、跑一個指令看結果——這些直接做，派 subagent 反而更慢更貴。

---

## 二、可用的 subagent 型別與怎麼選
> 以你 session 的 system-reminder「Available agent types」清單為準。常見且可靠的：

| subagent_type | 用途 | 能不能改檔 |
|---------------|------|-----------|
| `Explore` | 唯讀搜尋、摸清結構、定位程式碼 | ❌ 只讀 |
| `general-purpose` | 複雜多步、要動手做完整任務 | ✅ 可 |
| `Plan` | 設計實作計畫、架構取捨 | ❌ 只讀 |

不要假設有自訂 agent（如 `code-reviewer`）存在，除非清單裡有。

---

## 三、派工三件套（每次委派都要講清楚這三個，缺一不可）
1. **目標與動機**：要什麼結果、為什麼（讓 subagent 能自行取捨）。
2. **驗收條件**：怎樣算做完、有哪些硬性限制（例：「必須 `npm run build` 通過」「不准直接 insert orders」）。
3. **回報格式**：要它回什麼、多長（見「回報合約」）。

缺了會發生什麼（反例）：只說「看一下結帳流程」→ subagent 回 3000 字流水帳，你還要再讀一次，等於沒外包。

---

## 四、顯式指定 model 與 effort（⚠️ harness 限制，務必先確認你有哪些工具）

**先做一件事**：看你這個 session 的 system-reminder／工具清單，確認**你實際有哪些工具**。
不要照抄下面去呼叫一個你沒有的工具——那會變成撞牆重試。（諷刺提醒：這份制度初稿就曾假設有 `Workflow` 工具，被 fresh-context 審查抓到「你的清單裡沒有」。這正是「事實先驗」的活教材。）

**已知的真實能力**：
- **`Agent` 工具（幾乎一定有）**：可指定 `model`（`haiku`/`sonnet`/`opus`）。**不能指定 effort**。這是你調算力的主要手段。
- **`Workflow` 工具（不一定有——很多 session／subagent 都沒有）**：**只有**當你的工具清單裡確實列出它時才用；它的 `agent(prompt,{model,effort})` 能同時指定 model 與 effort（effort 的合法值以該工具當下 schema 為準，別背死）。**清單裡沒有就當它不存在，不要呼叫。**
- 主對話自己的 effort 由使用者 session 設定決定，你改不了。

**所以實務上（以一定存在的 `Agent` 為準）**：
- 一般委派 → 一律用 `Agent` 工具，靠**選 `model`** 調算力。
- 需要「更強的思考」→ **升 `model`（例如 opus）**，這就是本環境對 subagent 加算力的正解（Agent 無法指定 effort，別想靠 effort）。
- 除非你確認有 `Workflow` 且任務是多階段大量 fan-out，否則**完全忽略 Workflow**，不影響任何事。

**model 選擇對照（本環境：主用 Sonnet，高風險可升 Opus）**：
| 任務性質 | model | 例子 |
|---------|-------|------|
| 機械、唯讀、低風險 | `haiku` | 掃檔案列清單、跑格式化、批次套已驗證的改法 |
| 一般實作 / 搜尋 / 審查 | `sonnet` | 寫一個元件、Explore 摸結構、一般 code review |
| 高風險判斷 | `opus` | 資安/RLS/認證、金流、破壞性 DB 操作、架構決策、多答案評審選優 |

**高風險清單（碰到就升 Opus 或加第二意見）**：RLS/認證、金流與金額計算、`DROP`/無 `WHERE` 的批次寫、刪資料、改部署設定、任何「錯了會讓 A 看到 B 資料」的事。

## 五、回報合約（subagent 回什麼）
- **只回結論**：關鍵發現 + `檔案:行號`，不回過程、不貼長檔內容。
- **長產物落檔**：報告/程式碼/資料超過 ~30 行 → 寫成檔案，只回「路徑 + 一句摘要」。
- 建議在派工 prompt 結尾明寫：「回報 ≤15 行，格式：① 結論 ② 關鍵 `檔案:行號` ③ 還有什麼沒確認」。
- 需要結構化資料時：**若有 `Workflow`** 用它的 `agent(..., {schema})` 強制 JSON；**沒有**就用 `Agent`，在 prompt 明寫「只輸出這個 JSON 格式、不要其他文字」。

## 六、升降級路徑（不要在同一個坑裡燒 token）
- **Haiku 錯 1 次 → 直接升 Sonnet 重派**（別讓 Haiku 硬試）。
- **Sonnet 同一子任務連錯 2 次 → 升 Opus**，且派工時**附上完整失敗軌跡**（前兩次做了什麼、錯在哪、錯誤訊息），別讓 Opus 從零開始。
- **模式解出後降級**：一旦「正確改法」確定了，剩下的重複套用降回 `haiku`/`sonnet` 批次做（省錢）。
- **硬上限：同一件事最多重試 2 輪**。第 3 輪不要再試——改路（換方法）或停下來問使用者（見 `20-judgment-rubrics.md`）。

---

## 七、驗證不自驗（做完 ≠ 正確）
**寫的人不驗自己**。驗收一律派 **fresh-context** 的新 subagent（沒看過實作過程，才客觀）：
- **檔案類**：read-back 驗證（讀回來確認內容真的落地、完整、沒截斷）。
- **程式碼類**：跑測試或實跑（本環境 `npm run build` 必過；DB 相關用 `SUPABASE_ACCESS.md` 的 `SET ROLE` 實測）。
- **高風險判斷**：加第二意見（另一個 Opus agent 獨立審），或多答案評審（產 2–3 個方案，派一個 agent 評分選優）。

**誠實邊界**：拆解＋驗證＋多樣本評審能補「執行品質」；但**模糊題、品味與取捨判斷補不了**——遇到這種，升 Opus + 給使用者第二意見，或明說「這需要你決定」（見 `20-judgment-rubrics.md`）。

## 附錄 · Windows 環境雷區
- 兩個 shell：`Bash` 工具用 POSIX 語法、`PowerShell` 工具用 PS 語法，**不要混**。
- 路徑用正斜線 `/` 或跳脫 `\\`。
- **SQL / 多行 JS 一律寫檔再執行**，不要 `node -e "..."` 內嵌複雜引號（dollar-quote、`\u` 會爆）。
- 這環境的**瀏覽器預覽工具連不到本地 dev server**（網路命名空間不同步）→ 驗證前端改動靠 `npm run build` + 直連 DB 測資料，不要卡在預覽工具。
